# 5. Dependency Inversion Principle (DIP)

### Definition

A class should **depend on abstractions (interfaces), not concrete implementations**.

This reduces tight coupling and makes the code easier to extend, maintain, and test.

---

## ❌ Violation of DIP

Suppose we have an `APIClient` class that internally uses Playwright's `request` object.

```ts
import { request } from '@playwright/test';

class APIClient {

    constructor(method, body, headers, url) {
        request.method(url, {});
    }

}
```

Here, `APIClient` is tightly coupled to Playwright's `request` library.

If in the future we want to use another HTTP library like **Axios** or **SuperAgent**, we have to modify the `APIClient` implementation.

❌ This violates DIP.

---

## ✅ Following DIP

Instead of creating or directly using a specific API library inside the class, pass the API client (or its interface) as a dependency.

```ts
class APIClient {

    constructor(request, method, body, headers, url) {
        request.method(url, {});
    }

}
```

Now, `APIClient` is independent of the underlying API library.

We can pass a Playwright request object, Axios client, or any other implementation without changing the `APIClient` class.

This makes the code loosely coupled and easier to maintain.
